## Как объяснить ключевые решения

### "Почему guided flow, а не обычный лендинг?"

> Аудитория приходит из TikTok — короткое внимание, мобильный. Бесконечный скролл теряет людей. Guided flow ведёт шаг за шагом: каждый экран решает одну задачу и заслуживает переход на следующий. Конверсия выше, потому что пользователь не чувствует перегруз.

### "Почему Next.js App Router?"

> SSR для SEO и скорости. API routes для backend-логики без отдельного сервера. Middleware для access control. Server components для минимального JS на клиенте. Vercel для zero-config deploy. Всё в одном стеке — меньше сложности.

### "Почему MongoDB, а не PostgreSQL?"

> Гибкая схема хороша для контента (уроки, quiz, рекомендации). Не нужны сложные JOIN'ы. Mongoose даёт типизацию и валидацию. Для будущих AI-фичей (результаты диагностики, рекомендации) document model удобнее.

### "Почему не Redux / Zustand?"

> App Router даёт server components — большая часть данных загружается на сервере. Клиентское состояние минимально: flow navigation, quiz answers, video position. React hooks достаточно. Over-engineering state management для этого размера — антипаттерн.

### "Как продукт масштабируется?"

> Три уровня. Первый: новые продукты (Purchase.productSlug уже поддерживает). Второй: AI (отдельный домен lib/ai/, не трогает существующий код). Третий: мультитенант (другие эксперты на той же платформе — отдельная route group, CMS для контента).

### "Что бы вы сделали по-другому?"

> С опытом этого проекта — начал бы с CMS для контента вместо data.ts (быстрее менять без деплоя). И использовал бы Mux для видео с первого дня — YouTube embed ограничивает UX (нет resume playback, нет analytics).

## Что делает проект "senior-level"

Не объём кода. А:

1. **Продуктовое мышление** — от проблемы пользователя до архитектуры
2. **Domain separation** — код организован по бизнес-доменам, не по типам файлов
3. **Thin routes, thick lib** — API routes делегируют, не решают
4. **Consistent patterns** — один подход по всему проекту
5. **Access control** — middleware, не if/else в каждом компоненте
6. **Scalability** — новая фича = новый модуль, не правки 20 файлов
7. **Trade-off awareness** — понимаешь, что выбрала и почему
